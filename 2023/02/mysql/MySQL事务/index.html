MySQL事务

Posted on 2023-02-02

ACID

四大特性的实现原理

事务拥有四大特性，原子性、一致性、隔离性、持久性。解释如下：

原子性：事务的所有操作，要么都执行，要么都不执行

一致性：事务执行前后，数据库完整性没有被破坏，意思是指事务的操作不会破坏任何 MySQL 规则，如唯一索引约束等

隔离性：多个事务并行执行时，最终呈现的效果是串行的。意思是对于事务执行过程中，感知到的数据变化应该只有自己操作引起的，不存在其它事务引发的数据变化。

持久性：一旦事务提交，对数据库的修改将是永久的，不会因为系统故障等丢失

MySQL 如何保证四大特性

原子性：undo log

一致性：程序故障 + AID（？）

隔离性：undo log + 事务 ID + mvcc

持久性：redo log

隔离级别

各个级别的定义、可能存在的问题

多个事务并行执行就会产生各种奇怪的问题，事务隔离级别是为了解决这些问题，实现事务的隔离性产生的。

并行事务存在的问题有：

脏读：读取到了其它事务未提交的数据，事务 A 在事务 B 读取数据时刚好修改了数据，B 读取后 A 反而回滚了事务，就使得事务 B 读取到了本不应该存在的数据！

不可重复读：一个事务前后读取某数据，得到的结果不一致

幻读：事务前后读取的数据量不一致，幻读专指 insert 的情况

事务隔离级别有：

读未提交：事务未提交时，它的修改就可以被其它事务读取到，存在脏读问题

读提交：只有事务提交后，它的修改才会被其它事务读取到

可重复读：事务前后读取相同数据行，得到的信息前后一致。

串行化：对同一行记录而言，事务总是串行进行

Oracle、SQLserver、PostgreSQL 数据库的默认隔离级别都是读提交，MySQL 的默认隔离级别是可重复读。

查看事务隔离级别：

# mysql >= 5.7.20 是 transaction_isolation，之前和 mariadb 是 tx_isolation

mysql> show variables like ‘transaction_isolation’;

数据库为了实现以上数据库隔离级别，通过在事务执行前创建一个一致性视图，控制事务读取的逻辑结果。

读未提交时，不创建数据视图。

读提交时，数据视图在每个事务 SQL 执行时创建。

可重复读时，视图在事务启动时创建，整个事务操作都是用这个视图

串行化也没有数据视图，直接通过加锁的形式避免同时操作同一行

可重复读的实现

一致性视图和行锁从两个不同的方面保证了事务的原子性和隔离性。

可重复读的核心就是一致性读（consistent read）。首先，一致性视图使得事务读操作变为快照读，每次读取该行数据都是事务启动时的数据版本，而当事务内部更新了某个字段的数据，因为也会记录当前事务版本号（和 undo log）到行数据版本，所以修改后读取也符合预期。而行锁通过写写互斥的操作保证了一个事务内的写操作不会被其他事务中断。

而事务更新数据的时候，只能用当前读。如果当前的记录的行锁被其他事务占用的话，就需要进入锁等待。

使用可重复读的场景

当前查询操作不希望受到实时更新影响的时候

假设你在管理一个个人银行账户表。一个表存了账户余额，一个表存了账单明细。到了月底你要做数据校对，也就是判断上个月的余额和当前余额的差额，是否与本月的账单明细一致。你一定希望在校对过程中，即使有用户发生了一笔新的交易，也不影响你的校对结果。

这时候使用“可重复读”隔离级别就很方便。事务启动时的视图可以认为是静态的，不受其他事务更新的影响。

长事务的弊端

长事务会导致系统中大量一致性视图、undo log 和锁资源无法释放

长事务是指启动和提交中间操作很多，间隔久的事务。长事务会导致系统存在大量 read-view 和 undo log，占据大量内存空间，除此以外长事务还会占用锁资源，严重威胁并发性甚至可能拖垮数据库。

事务启动方式推荐

建议使用 commit work and chain 启动事务，指 autocommit = 1 情况下使用 begin 显示启动事务，然后执行 commit work and chan 代替单纯的 commit 提交事务，此操作可以在提交事务的同时自动启动下一个事务，减少了一次 begin 事务的交互。

面对事务的优化

优化的角度可以有很多：

  1. 从监控的角度可以
    1. 通过 information_schema.innodb_trx 监控事务的持续时间
    2. 通过 pt 工具监控事务的持续时间
  3. 从配置角度可以
    1. 增加 undo log 空间，将 innodb_undo_tablespace 设置为 2 或更大的值
    2. 通过配置 max_execution_time 控制事务最大执行时间
  5. 另外还可以想办法减少长事务的执行时间
    1. 从优化语句的角度入手，比如优化索引
    2. 也可以将不必要的大批量查询导致的长事务改为分批的小查询

MVCC

mvcc 全称多版本并发控制。每个事务启动时（RR）生成一个一致性视图，通过一致性视图声明：承认在事务启动前的数据版本（和事务自己的修改），不承认事务启动后数据发生的改变。

一致性视图

MVCC 在实现上， InnoDB 为每个事务构造了一个数组，用来保存这个事务启动瞬间，当前正在“活跃”的所有事务 ID。“活跃”指的就是，启动了但还没提交。数组里面事务 ID 的最小值记为低水位，当前系统里面已经创建过的事务 ID 的最大值加 1 记为高水位。这个视图数组和高低水位，就组成了当前事务的一致性视图（read-view）。

这样，对于当前事务的启动瞬间来说，每个数据版本的 row trx_id，有以下几种可能：

  1. 小于等于低水位事务 ID，表示这个版本是已提交的事务或者是当前事务自己生成的，这个数据是可见的；
  2. 大于高水位事务 ID，表示这个版本是由将来启动的事务生成的，是肯定不可见的；
  3. 在高低水位事务 ID 之间，那就包括两种情况：
    1. 若 row trx_id 在事务 ID 数组中，表示这个版本是由还没提交的事务生成的，不可见；
    2. 若 row trx_id 不在事务 ID 数组中，表示这个版本是已经提交了的事务生成的，可见。

当前读

因为数据库更新当然是在最新版本基础上更新，但是仅用一致性视图的话就会导致可能在之前的某个版本基础上更新！

为此，大佬提出了新的规则：更新数据都是先读后写，而且这个读是当前读！

此外，select 语句通过加锁也可以做到当前读：

mysql> select k from t where id=1 lock in share mode;

mysql> select k from t where id=1 for update;
