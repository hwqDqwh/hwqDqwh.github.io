MySQL锁

Posted on 2023-02-01 Edited on 2023-02-03
浅析 MySQL 锁

数据库在应对并发访问时需要一套规则合理控制资源访问，而锁就是用来实现控制规则的数据结构。

全局锁

使用场景、跟只读的区别

顾名思义，全局锁是对整个数据库实例加锁。

通过以下语句可以施加全局锁（读锁）：

Flush tables with read lock

全局锁的使用场景

全局锁的经典使用场景就是【全库逻辑备份】，也就是把整库查询出来存成文本。

全库备份为什么要施加全局锁？

发起数据库备份时，不同表备份顺序的先后会有时间差，在这个时间差之内对表的操作就有可能造成数据不一致，从而引发事故。也就是说数据备份得到的库是逻辑不一致的。

mysqldump 的导出

官方自带的逻辑备份工具是 mysqldump。当 mysqldump 使用参数 -–single-transaction 的时候，导数据之前就会启动一个事务，来确保拿到一致性视图。而由于 MVCC 的支持，这个过程中数据是可以正常更新的。

当然，前提是表对应的存储引擎要支持事务，MyISAM 就不支持，也是它没落的原因。

全局锁和设置只读的区别

看起来全局锁的使用场景也可以通过设置数据库只读（set global readonly = true;）来代替，但是他们中间还是有区别的：

  1. readonly = true 被广泛使用在业务系统中的其它逻辑，比如用来判断主从，使用它可能导致其它问题
  2. 异常处理机制上不同，施加全局锁后如果发生异常（如客户端断开），全局锁会被自动释放；而 readonly 则会维持。

表级锁

MySQL 里面表级别的锁有两种：一种是表锁，一种是元数据锁（meta data lock，MDL)。

注：表级锁不会死锁。（？）

表锁

触发时机

表锁的语法是 lock tables … read/write。与 FTWRL 类似，可以用 unlock tables 主动释放锁，也可以在客户端断开的时候自动释放。需要注意，lock tables 语法除了会限制别的线程的读写外，也限定了本线程接下来的操作对象。

当某个事务需要查询某个表的大部分或全部数据时，如果使用更细的行锁会导致查询效率更慢，而且容易引长时间发锁等待或死锁等，此时考虑使用表锁提高事务执行效率。

元数据锁

解决什么问题、使用方式、DDL 的正确方式

MySQL 中 DDL 不属于事务范畴，如果 DDL 和事务同时修改某个表时可能破坏事务特性等问题，为了解决这类问题，MySQL 5.5.3 引入了元数据锁（metadata lock），缩写 MDL。

MDL 的作用是解决同一张表上事务和 DDL 并行执行可能破坏数据不一致的问题。例如一个查询正在遍历一个表中的数据，而执行期间另一个线程删除了该表某一列，就会导致查询线程拿到的结果跟表结构对不上

MDL 不需要显式使用，当对一个表做增删改查操作的时候，加 MDL 读锁（？）；当要对表做结构变更操作的时候，加 MDL 写锁。

MySQL 5.6 支持了 online DDL。

MDL 会直到事务提交才释放，所以 DBA 要尽量在闲时在做表结构变更，而开发则应尽量保证事务效率，避免大事务。否则容易引发大规模锁等待。

问：备份一般都会在从库上执行，你在用–single-transaction 方法做逻辑备份的过程中，如果主库上的一个小表做了一个 DDL，比如给一个表上加了一列。这时候，从备库上会看到什么现象呢？

答：备份时会启动全局锁，整个库处于只读状态。 主库对表做DDL，该表会启动写锁，该表不能读写。此时备库中应该看不到 DDL 的操作结果。

安全的改表结构方式

如果你要变更的表是一个热点表，虽然数据量不大，但是上面的请求很频繁，而你不得不加个字段，你该怎么做呢？ 这时候 kill 可能未必管用，因为新的请求马上就来了。比较理想的机制是，在 alter table 语句里面设定等待时间，如果在这个指定的等待时间里面能够拿到 MDL 写锁最好，拿不到也不要阻塞后面的业务语句，先放弃。之后开发人员或者 DBA 再通过重试命令重复这个过程。

ALTER TABLE tbl_name NOWAIT add column …

ALTER TABLE tbl_name WAIT N add column …

自增锁

自增锁是 InnoDB 是在 auto_increment 时保证该值正确的进行自增的锁。

自增锁是执行完立刻就释放资源的锁！

行锁

锁索引与降级、两阶段锁

MySQL 的行锁是有存储引擎自主实现的，不是所有引擎都支持行锁，例如 MyISAM。

行锁因为涉及 MVCC 数据快照等保存操作，开销比表锁大，但因为粒度小姑依然对提高并发有帮助

注意：Innodb 行锁锁的是索引， 如果更新、删除条件没走索引， 会降级成表锁

两阶段锁

定义、目的、优化

在 InnoDB 事务中，行锁都是两阶段锁：

  1. 锁在需要的时候才施加
  2. 锁的添加和释放分步进行，中间不允许交叉施加和释放锁。
  3. 事务提交的时候释放锁资源

如何通过减少锁冲突来提升业务并发度

如果事务中需要锁多个行，要把最可能造成锁冲突、最可能影响并发度的锁尽量往后放。从而利用【锁是在需要的时候才施加的】的特性，减少数据行被锁的时间。

死锁

处理策略、

当并发系统中不同线程出现循环资源依赖，导致这几个线程都进入无限等待的状态，称为死锁。但是哪怕优化减少了锁冲突也不可能完全避免。

死锁处理策略

当出现死锁以后，有两种策略：

一种策略是超时退出。死锁时直接进入等待，直到超时。这个超时时间可以通过参数 innodb_lock_wait_timeout 来设置。弊端是如果 innodb_lock_wait_timeout 设置很大在实际业务场景中是不可接受的，如果设置很小则容易误伤只是正常锁等待的语句。

另一种策略是发起死锁检测。发现死锁后，主动回滚死锁链条中的某一个事务，让其他事务得以继续执行。将参数 innodb_deadlock_detect 设置为 on，表示开启这个逻辑。

死锁监测是常用的策略。

但是死锁监测是要耗费大量的 CPU 资源的。整体死锁检测的耗费代价是 O(n^2)级别。 并发更新同一行的1000个线程，整体耗费的死锁检测操作为1000*1000=100万。 为什么是个乘法——并发更新此行R1的某单个线程Tx，其所作的死锁检测工作为，Tx会查看自身持有的行锁，同时遍历其他999个线程所持有的行锁，总共为1+999=1000次。 为什么会遍历其他999个线程，而不是仅看当前持有R1行锁的这个线程就行了？—— 因为行锁排队。某线程Tm排队获取R1行锁，排在Tx前。如果Tx当前持有行锁R2，过会Tm先于Tx获持R1后，会变成——Tm持有R1，等待R2 && Tx持有R2，等待R1——Tm和Tx成环死锁。 因此并发更新同一行的有N个线程，对应的死锁检测耗费代价为O(N^2) 。死锁检测不可避免，为防止死锁检测代价过高引起性能问题，应该想办法减少同时对同一行更新的并发度，即降低N值。

优化方法：

  1. 如果能确保业务不会出现死锁，可以临时把死锁监测关掉。
  2. 控制并发度，基本思路是，对于相同行的更新在进入引擎之前排队，这样 innodb 内部就不会有大量的死锁监测工作了。
  3. 从业务层面优化，例如做到将一行拆成多行的方式存储，以此减少行锁竞争。

读锁和写锁

读锁和写锁是从另一个视角看到锁定义的，读写锁施加在行级数据时又叫行锁，施加到表级数据时又叫表锁。

读锁又叫共享锁，大家都能读，同时大家都不能写。自身写会立刻报错，别的线程写则会锁等待。

写锁又叫排它锁，只有我能读写，其他人都不能读写。别的线程读写会锁等待。
