问题

最常见的问题有两处，都是缺乏证书导致的

  1. 下载安装文件时无法下载
    1. 改成使用 http 协议下载即可 php -r “copy(‘https://getcomposer.org/installer‘, ‘composer-setup.php’);”
    2. 或者，使用 wget 下载 composer-setup.php
      wget –no-check-certificate https://install.phpcomposer.com/installer -O ./composer-setup.php
    3. 或者 curl -sS https://getcomposer.org/installer | php
  2. 安装（下载 composer.phar）时过不了验证，提示 The “https://getcomposer.org/versions“ file could not be downloaded: SSL operation failed with code 1.
    1. 使用 php composer-setup.php –disable-tls 代替 php composer-setup.php 执行

总结一下各种情况下主从故障的恢复

binlog 未正常关闭

某一次公司机房断电后，因为 MySQL 服务未正常关闭，导致主从 binlog 不一致，事故持续半小时。

解决思路是，因为这里都是实时数据类的数据，所以准备让从库跳过所有异常追上主库，然后在主库删除半小时内新增的所有数据，让实时任务重新 insert 数据，最终完成主从数据一致。

增加下面一行

重启从库数据库服务。

等待数据库追上主库的 position。

主库执行

检查主从执行和数据情况，确保从库执行了delete 语句。

修改 my.cnf 关闭跳过异常，重启从库服务。

最后检查定时任务，确保实时任务将当日数据重新 insert，数据就补上了。

这个操作的弊端就是会导致主库方面出现一段时间的数据缺失，同时操作存在出错导致 delete 范围不符合预期的可能。总的来说是针对实际情况的选择，不宜作为常规方案。

大致意思是，如果在使用 systemd 时，可以打开的文件描述符数量不能超过 LimitNOFile，如果没有设置则参考 DefaultLimitNOFILE。同时 open_files_limit 不能设置为大于 ulimit -n 的数量。

公司在内网部署了专门的 gitlab 服务用于代码管理，然后发现在开启 VPN 的时候总是无法连接内网 gitlab 服务，找了很久才找到原因，以下是bl办法。

打开 VPN 的配置文件，将下述 dns.enable 改为 false 然后保存，在 VPN 上重载配置文件即可。

原因大概就是内网 gitlab 服务本身配置了 dns 解释，然而 VPN 自身也启用 VPN 的话就存在互相覆盖的问题。

  • pstree: 类似 ps 命令的进程查看工具，可以将进程以树形结构可视化输出。部分内核或挂载信息会被以 ? 代替
