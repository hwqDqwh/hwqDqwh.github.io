MySQL 索引学习整理

Author: hwqDqwh Date: 2022-03-15 Reading Time:11.6 mins words:3267 Category: MySQL Views: -

索引是为了提高查询效率组织的数据目录。

几种数据结构组织的索引

索引作为加速查询效率的工具，其组成数据结构尤为重要，以下分析不同数据结构构建的索引效率并尝试体现其迭代优化的过程。
首先是非常常见的集中数据结构：哈希表、数组、二叉树，先大概看下它们构建索引的优劣：

哈希表

哈希表的思路很简单，使用 key-value 结构存储数据，通过哈希函数和 key 值计算 value 的存储位置，直接读取。
遇到多个 key 计算哈希值结果一致的情况，就将多个 value 以链表的形式存储对应位置上。

哈希表的特点是：

  1. 利用哈希底层是数组，而数组随机访问快（O(1)）的特点加速查询效率，故等值查询很快
  2. 哈希表中各个元素之间是无序的，所以插入很快，只需要末尾直接添加即可；但是范围查询很慢，因为需要全表扫描

有序数组

有序数组在等值查询和范围查询的效率都很高（二分法，O(logN)）但是插入效率很低（往某位置插入数据，需要移动后面所有的数据元素）。
因为插入效率慢，所以有序数组只适合静态数据的存储引擎

平衡二叉查找树

平衡二叉查找树有以下特点（详见算法笔记）：

  1. 每个节点的值大于其左子节点值，小于右子节点值
  2. 最后一层叶子节点靠左排列
  3. 所有叶子节点高度差 <= 1

得益于数据结构，构建的索引查询和插入更新的效率都是 O(logN)，是理想的效率模型。
但是平衡二叉查找树实际上依然有很大限制！就是二叉树因为每个节点只有 2 个子节点，同时每个节点值存储一个数据。由此导致存储和读写的 IO 消耗很高，假设某表有 100W 节点，此时树高为 20，那么每次等值查询需要访问 20 个节点数据进行判断，即使二叉搜索树存储在内存中也无法接受。

为了解决优化存储数据低效的问题，大佬们提出了 B 树。

B 树

B 树是在平衡二叉查找树的基础上提出的，主要解决两个问题：
问题一：二叉树的每个节点只有 2 个子节点
解决：B 树是 N 叉树，根据特定规则（？）在插入等操作时进行节点分裂、合并，总体上维持较好的平衡，同时 N 叉树导致树高大大降低（innodb 的 N 约为 1200）。

问题二：二叉树每个节点只能存储一个数据
解决：B 树每个节点同时保存多个元素，此时每个节点对应磁盘上的磁盘块，也就是说根据磁盘块来组织 B 数的节点存储。这样也主动适应了【磁盘读取每次都是读取一个磁盘块数据】的特性

B 树优化了很多磁盘操作层面的问题，不过大佬们依然不够满意，原因就是因为每个节点对应磁盘上一个磁盘块的情况下，受限于磁盘块大小固定（4k 或 16k），而每个磁盘块需要同时存储关联节点的指针和数据信息，导致 N 叉树的 N 还是没有很大，也就是说树高还是蛮高的，由此就有了 B+ 树

B+ 树

B+ 树在 B 树的基础上进一步优化了节点的存储问题，简单说就是 B+ 树只有叶子节点存储数据，非叶子节点只存储索引。同时 B+ 树的数据是顺序存储，提高了范围查询、排序查询的效率。
通过仅在叶子节点存储数据，在每个磁盘数据页大小为 16K 的条件下做到 1000-1200 叉树，此时存储 10 亿条键值数据层高只有 3，只需要 3 次磁盘 IO 即可得到结果（事实上根节点通常常驻内存，只需要 2 次 IO）
B+ 树通过在叶子节点将构建双向链表，叶子节点内部的数据构建单向链表，实现了数据顺序存储，提高查询效率。

B+ 树因为读写性能优秀，同时适配磁盘的访问模式，已经广泛应用在数据库引擎的存储结构上。

注：不同存储引擎对 B+ 树的实现、使用有细微差别，例如 MyISAM 中 B+ 数的叶子节点不直接存储数据，而是存储数据的文件地址，innodb 则是直接存储数据。

innodb 的索引模型

innodb 中每个数据表有一棵主键索引树和若干棵非主键索引树，数据结构均为 B+ 树。

主键索引树：innodb 的主键索引树的叶子节点存储的是具体的行数据，非叶子节点则存储主键值。
非主键索引树：叶子节点存储主键值，所以使用时都是在非主键索引树查询得到对应的主键值，然后到主键索引树查询对应行数据（这个过程叫回表）。
查询时，依次优先使用主键索引树 > 非主键索引树 > 不使用索引（直接遍历主键索引树）

主键索引又叫聚簇索引，非主键索引又叫二级索引（非聚簇索引）。

innodb 索引维护

因为 B+ 树本身维护了索引的有序性，那么就需要考虑增删对索引树结构的影响了，因为插入新记录和删除记录都可能引发索引数的叶子节点分裂或合并（比如新记录使得某个磁盘页用满，就需要组织新磁盘页存储数据，此时体现在 B+ 树上就是新的叶子节点）。

innodb 为了优雅地解决索引树叶子结点的维护问题，使用了以下办法：

  1. 提倡使用独立的自增值作为表主键，由此有以下好处：
    1. 自增主键使得主键索引树在新插入记录时总是【追加】操作，几乎不费力就能维护顺序结构
    2. 自增值代替业务字段作为主键可以节省存储空间
  2. 删除时通过逻辑删除而不是物理删除方式页合并
    1. 在删除操作时对删除记录进行标记，因为不是真正的删除操作所以不会触发页合并
    2. 在特定的时候触发页合并

使用业务字段作为主键的坏处

  1. 业务字段长度可能不可控而且更长，会导致非主键索引树的存储需要更大空间（叶子结点数据是主键值）
  2. 业务字段通常不是递增的，会导致新增主键值在主键索引树不再是追加，大量页分裂使得性能下降。

重建索引

重建普通索引树很简单：

alter table table_a drop index k;
alter table table_a add index k(`k`);

通过重建索引会得到一个全新的索引树，优化旧索引树因为页分裂等导致的数据空洞问题，提高数据页的使用率，由此节省了空间。

但是对主键的操作应该更谨慎，如果删除主键列/主键索引，MySQL 会使用内置的 RowID 代替原主键构建索引树（包括二级索引树），表对应的索引树都面临重新构建的需要。
正确的做法是：

alter table table_a engine=InnoDB;

索引树的使用

使用索引树查询时，会对数据挨个进行搜索，直到搜索到不符合要求的数据停止。

根据 MySQL 45 讲大佬的举例解释一下：

mysql> create table T (
ID int primary key,
k int NOT NULL DEFAULT 0, 
s varchar(16) NOT NULL DEFAULT '',
index k(k))
engine=InnoDB;

insert into T values(100,1, 'aa'),(200,2,'bb'),(300,3,'cc'),(500,5,'ee'),(600,6,'ff'),(700,7,'gg');

解释语句 select * from T where k between 3 and 5 的执行过程：

  1. 搜索 k 索引树查询得到 k=3 记录，获得 ID=300
  2. 到主键索引树搜索 ID=300 的记录，返回数据给执行器
  3. 通过磁盘指针，继续查询 k 索引树中的数据，得到 k=5 对应 ID=500。注意这里不需要重新遍历 k 索引树，因为索引是有序的
  4. 再次回表，搜索主键索引树得到 ID=500 航记录，返回数据给执行器
  5. 重复 3 步骤，得到下一个值 k=6，不满足查询条件，查询结束

综上，查询的过程中查询了 k 索引树的 3 条记录，回表了 2 次。此时 binlog 中的 row_examined 应该是 3

如果查询语句是 select ID from T where k between 3 and 5，因为 k 索引树可以直接得到 ID 值，此时就不需要回表了。此时我们叫 k 索引为覆盖索引。
显而易见，覆盖索引减少了回表的次数，所以常常作为优化的手段之一。

覆盖索引和冗余索引

覆盖索引：创建联合索引覆盖高频查询条件，达到提高查询速度的目的

上面已经说了所谓覆盖索引就是直接将需要涉及查询的字段建立联合索引，通过减少回表操作提高效率。
而冗余索引则是相反，指多个索引的前缀列相同，互相造成资源冗余浪费的情况。

覆盖索引和冗余索引的考量是个矛盾的过程，总得来说就是综合考虑，根据实际情况使用。假设有两列，比如"公民身份证号"和"姓名"，虽然姓名一般看上去不需要加入索引，但是事情不是绝对，如果有高频请求总是查询这两个字段建立覆盖索引能显著加速则是 OK 的；相应的，如果有几个查询都用到了"身份证号"和其它字段，而且频率不高，无脑构建覆盖索引反而导致资源浪费（因为联合索引本身就意味着更大的存储空间和维护成本）。

所以一般从几个方面综合考虑：

  1. 查询的频率
  2. 组织覆盖索引造成的资源使用问题

最左前缀原则

联合索引的字段（和字符）按照定义顺序，从左到右依次生效

为了避免创建多个索引，B+ 树支持最左前缀原则，就是多个字段的联合索引可以允许前 N 个字段索引生效。原因是存储引擎构建联合索引树对索引值排序时，先根据第一个字段排序，当第一个字段值相同时再对第二个字段排序，以此类推构成有序索引，再存储到索引树的非叶子节点上。

注：最左前缀原则不仅仅指联合索引的最左 N 个字段，也可以是某个字符串字段的左边 M 个字符。

问：怎么确定某个查询在使用了最左前缀原则时，使用到了多少个字段和某个字段的前几个字符呢
答：我目前理解都是根据具体查询语句决定，先尽量使用左边 N 个字段，再考察字段的左边 M 个字符

索引下推

MySQL >= 5.6 搜索联合索引树得到索引信息时，会优先判断索引字段是否符合查询条件，不符合则直接跳过，减少了回表次数

以市民表的联合索引（name, age）为例。如果现在有一个需求：检索出表中“名字第一个字是张，而且年龄是 10 岁的所有男孩”。那么，SQL 语句是这么写的：

mysql> select * from tuser where name like '张%' and age=10 and ismale=1;

根据最左前缀原则，MySQL 会一直向右匹配索引，直到遇到范围查询（>、<、between、like）就停止。所以这条查询只会使用到 name 字段的索引。
在 MySQL < 5.6 时， 此时会根据 name 字段索引筛选得到的主键值回表，取出行记录中的 age 和 ismale 进行判断，返回符合判断条件的记录。
MySQL >=5.6 进一步优化，因为有索引（name，age）存在，就会根据联合索引最左前缀原则拿到的索引信息，先判断索引中包含的字段是否符合条件（这里是 age，判断是否等于 10），符合时才会回表，以此减少回表次数，这就叫索引下推。
