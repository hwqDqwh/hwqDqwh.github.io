APScheduler 理解

介绍

APScheduler 全称 Advanced Python Scheduler，是一个开源 python 库。利用 APScheduler 可以在程序中管理任务并控制任务按指定间隔形式执行。另外 APScheduler 可以作为跨平台任务调度的替代品，借助程序代码实现跨平台使用、迁移的目的。
需要注意的是，官方 github 有提示 APScheduler 本身不是一个守护进程，也不附带任何命令行工具。所以想要定时执行需要自主考虑好【守护】的问题，笔者自己初学就很疑惑【怎么做到定时启动执行】的问题，其实这个【定时启动】不是 APScheduler 本身考虑的问题了。

APScheduler 作为一个任务调度器其实感觉跟自己给公司项目写的有挺多共通之处（(￣▽￣)／），当然自己的项目无论从代码到设计都任重道远。。

用法简介

网上关于 APScheduler 的使用有官方文档，博客教程也很多就懒得赘述了。如果不想看官方英文文档，个人推荐一个腾讯云的文章（见文末），这里只做简短的归纳。

触发器

APScheduler 有三种触发器模式，分别如下：

  • date: 适用于某个时间节点执行的仅一次任务
  • interval: 以固定时间间隔执行某任务
  • cron: 类似 Windows 和 linux 自带的 crontab 模式

调度器

APScheduler 拥有好几种调度器，分别应用在不同场景：

  • BlockingScheduler: block 是锁的意思，顾名思义就是阻塞式的任务执行。官方文档描述是 use when the scheduler is the only thing running in your process. 当进程调度是你的进程中唯一需要执行的任务时使用
  • BackgroundScheduler: 将任务放在后台，跟命令行加 & 类似的效果，程序继续往下执行
  • 剩余的几种都是针对我们需要特定库的情况下使用，分别是：
    • AsyncIOScheduler 在我们需要使用 asyncio 库时使用；
    • GeventScheduler 对应 gevent（基于协程的网络库，额没用过）；
    • TornanoScheduler 对应 Tornano（一种 Python web 框架）；
    • TwistedScheduler 对应 twisted（一种事件驱动的网络编程框架，支持众多协议）；
    • QtScheduler 对应 Qt（这个 Qt 应该是对应 PyQt，一个基于 C++ Qt 的图形界面库）

任务存储

都是很熟悉的 SQL 和 noSQL 等了，APScheduler 支持 memory、redis、mongodb、SQLAlchemy 这几种按需取用即可

执行器

默认线程池，也可以使用进程池

APScheduler 和 crontab

APScheduler 和操作系统自带的原生 crontab 之间，乍一看不是一回事吗，其实不然：

  • 最核心的，APScheduler 依赖程度调起的任务管理，crontab 是由系统的守护进程 crond 提供的服务
  • APScheduler 支持线程级别的调度，crontab 是进程级别
  • APScheduler 作为语言库在应用层面提供了更多支持，如：
    • APScheduler 支持更多调度模式，兼容 crontab 外提供固定时间执行（date）和间隔时间后执行（interval）两种模式
    • 支持各种任务调度的管理操作，包括异常重启后自动执行本应得到执行的任务
    • 通过 AsyncIOScheduler 等支持了异步 IO 等深入的使用场景
    • 而这些 crontab 作为系统级服务没有支持，需要自主开发
      总的来说 APSCheduler 是一个应用层面的语言库，crontab 是系统提供的服务，两者有些共通点但是立意其实是不同的。

安装 APScheduler 遇到的一个问题

安装 apscheduler 时候报错

WARNING: pip is configured with locations that require TLS/SSL, however the ssl module in Python is not available.

自己的环境是 Python 3.7.9 + OpenSSL 1.1.1c FIPS 28 May 2019，安装命令如下：

pip3 install apscheduler

谷歌各种说法都有，包括 conda 环境的问题，刚刚好自己就在用就跑去检查环境了，查了一圈好像没啥问题，然后我在想是不是自己的 Python 版本跟 OpenSSL 搭配的问题，没找到直观的版本可用映射表，不过从 Python 官网 看到：

Currently Python versions 3.6 to 3.9 are compatible with OpenSSL 1.0.2, 1.1.0, and 1.1.1. For the most part Python also works with LibreSSL >= 2.7.1 with some missing features and broken tests.

乍一看自己使用 Python 3.7.9 + openssl 1.1.1 应该是完全 OK 的，不过转念一想，是不是应该翻一下 PEP 呢？
啊这不就在 3.7.9 Changelog 找到一个合理解释嘛。Python 3.7.9 的迭代日志中提到 3.7.8 的时候就将 openssl 支持叠加到 1.1.1f，对应的 issue 是 issue40146（Update OpenSSL to 1.1.1f in Azure Pipelines.）和 issue40162（Update Travis CI configuration to OpenSSL 1.1.1f.）
结合自己是 1.1.1c，解决方案很简单--只要升级 openssl 就好

yum upgrade openssl

最后升级 OpenSSL 到 OpenSSL 1.1.1k FIPS 25 Mar 2021 再安装 APScheduler 就可以了。
其实这个问题并不难解决，不兼容的情况下通常都会尝试下升级最新版本，不过知其然才应该是我们追求的，这次排查告诫自己要多看 PEP

思考

上述问题中最后发现是 openssl 版本的问题，在 python 迭代日志中看到开发者们修改了 Asure 和 Travis 的 yml 文件的版本关联，那么开发者们为什么要做此修改，这两个库版本又是怎么在 python 中发挥作用的呢？
可能需要深入阅读了解 python 源码了，希望自己能尽快作出回答
